test_ids

An Origen plugin to automatically assign and maintain test program bin and test numbers.

This plugin integrates with OrigenTesters and allows you to assign a range of numbers, or a function, to generate bin, softbin, and test numbers whenever you generate a test program with Origen. Unlike similar algorithms that may be implemented within an application, test_ids will maintain a record of the numbers assigned to each test so that they will stick even if the test flow order changes.

Integration

Simply add the gem to your application via the Gemfile. If your application is a plugin, then add it to your .gemspec and require it somewhere (at the top of your test program interface would be a good place):

<span class="line-numbers"><a href="#n1" name="n1">1</a></span>require <span style="background-color:hsla(0,100%,50%,0.05)"><span style="color:#710">'</span><span style="color:#D20">test_ids</span><span style="color:#710">'</span></span>

All that is required is then to configure the plugin in your interface’s initialize method:

Then anytime that you call flow.test(my_test_instance, options) within your application’s interface, assignments for :bin, :softbin and :number will automatically be injected into the options before it hits OrigenTesters. If an entry for any of the keys is already present in the options, then that will be given priority and TestIds will not attempt to assign a value for that attribute.

If you want to prevent TestIds from generating a given attribute and really pass nil for that attribute to OrigenTesters, then assign it to the value :none:

<span class="line-numbers"><a href="#n1" name="n1">1</a></span>flow.test my_test, <span style="color:#606">bin</span>: <span style="color:#A60">:none</span>   <span style="color:#777"># Assign no bin but allow TestIds to generate a softbin and test number</span>

A method is also provided to directly assign/retrieve the numbers for a given test from TestIds, this should be supplied with the same arguments that you would normally pass to flow.test:

<span class="line-numbers"><a href="#n1" name="n1">1</a></span><span style="color:#036;font-weight:bold">TestIds</span>.allocate(my_test, options)   <span style="color:#777"># => { bin: 5, bin_size: 1, softbin: 1250, softbin_size: 1 number: 10250010, number_size: 1 }</span>
<span class="line-numbers"><a href="#n2" name="n2">2</a></span>
<span class="line-numbers"><a href="#n3" name="n3">3</a></span><span style="color:#777"># The above returns the same numbers that would be injected into the options when calling:</span>
<span class="line-numbers"><a href="#n4" name="n4">4</a></span>flow.test my_test, options

If you want to prevent TestIds from tracking/assigning to a given test you can supply a no-track option:

<span class="line-numbers"><a href="#n1" name="n1">1</a></span>flow.test my_test, <span style="color:#606">bin</span>: <span style="color:#00D">10</span>, <span style="color:#606">test_ids</span>: <span style="color:#A60">:notrack</span>

Configuration

The various ID types are generated in the following order which places some constraints on the configuration options available to each one:

  • bin - These are generated first, and therefore can only be configured to be generated from a range. They can also be generated from a specific range per test. Please see Using specific ranges below.
  • softbin - These come next, and they can be generated from a range or from either a template or a function which references the bin number. They can also be generated from a specific range per test. Please see Using specific ranges below.
  • number - The test number comes last, and they can be generated from a range or from either a template or a function which references the bin number and/or softbin number. They can also be generated from a specific range per test. Please see Using specific ranges below.

Assigning a Range

All ID types can be given an independent range of numbers to pick from like this (where TYPE is bins, softbins or numbers):

<span class="line-numbers"><a href="#n1" name="n1">1</a></span>config.TYPE.include << <span style="color:#00D">3</span>           <span style="color:#777"># Number 3 only</span>
<span class="line-numbers"><a href="#n2" name="n2">2</a></span>config.TYPE.include << (<span style="color:#00D">100</span>..<span style="color:#00D">200</span>)  <span style="color:#777"># Also includes numbers from 100 to 200</span>

This can be called multiple times for the same ID type if your allocation is comprised of a non-contiguous range.

Certain numbers within the include allocation can be excluded like this:

<span class="line-numbers"><a href="#n1" name="n1">1</a></span>config.TYPE.exclude << <span style="color:#00D">150</span>         <span style="color:#777"># This is already assigned to something else</span>
<span class="line-numbers"><a href="#n2" name="n2">2</a></span>config.TYPE.exclude << (<span style="color:#00D">190</span>..<span style="color:#00D">195</span>)  <span style="color:#777"># A range is also acceptable</span>

When numbers are being assigned from a range, they will be selected based on an increment of 1 by default. However, in cases where you want multiple numbers to be reserved by each test you can change the default like this:

<span class="line-numbers"><a href="#n1" name="n1">1</a></span>config.TYPE.size = <span style="color:#00D">5</span>   <span style="color:#777"># Reserve 5 numbers for each test by default</span>

Then for a given range of 100 to 200 say, it would be assigned as 100, 105, 110, etc.

It is possible to override the size for an individual test by passing one or more of the options shown in the example below:

<span class="line-numbers"><a href="#n1" name="n1">1</a></span>flow.test my_test, <span style="color:#606">bin_size</span>: <span style="color:#00D">2</span>, <span style="color:#606">softbin_size</span>: <span style="color:#00D">10</span>, <span style="color:#606">number_size</span>: <span style="color:#00D">100</span>

Assigning a Template

Softbin and test numbers can also be generated from a template. Note that if you supply a template for a given ID type then you cannot also supply a range.

The template describes the form that the given number should have, using the following key:

  • b - A bin number digit
  • s - A softbin number digit
  • x - A uniqueness counter
  • 0-9 - A static digit

For example, to make the softbins equal to the bin number * 100, you can simply do:

<span class="line-numbers"><a href="#n1" name="n1">1</a></span>config.softbins = <span style="color:#A60">:bbb00</span>

For bins 1, 2, 3 this would generate softbins 100, 200, 300.

Suppose that you have many tests assigned to the same bin number and you would like the softbins to be unique. In that case you can make some of the digits a unique counter instead:

<span class="line-numbers"><a href="#n1" name="n1">1</a></span>config.softbins = <span style="color:#A60">:bbbxx</span>

If all of your tests were bin 3 for example, then this would assign softbins 300 to 399.

Assigning a Function

If your softbins or test numbers are a function which cannot be expressed using the above template rules, then you can always fall back to a custom function.

<span class="line-numbers"> <a href="#n1" name="n1">1</a></span><span style="color:#777"># The softbin function has access to the bin number</span>
<span class="line-numbers"> <a href="#n2" name="n2">2</a></span>config.softbins <span style="color:#080;font-weight:bold">do</span> |bin|
<span class="line-numbers"> <a href="#n3" name="n3">3</a></span>  bin * <span style="color:#00D">3</span>
<span class="line-numbers"> <a href="#n4" name="n4">4</a></span><span style="color:#080;font-weight:bold">end</span>
<span class="line-numbers"> <a href="#n5" name="n5">5</a></span>
<span class="line-numbers"> <a href="#n6" name="n6">6</a></span><span style="color:#777"># And the test number function has access to the softbin too</span>
<span class="line-numbers"> <a href="#n7" name="n7">7</a></span>config.numbers <span style="color:#080;font-weight:bold">do</span> |bin, softbin|
<span class="line-numbers"> <a href="#n8" name="n8">8</a></span>  (softbin * <span style="color:#00D">10</span>) + bin 
<span class="line-numbers"> <a href="#n9" name="n9">9</a></span><span style="color:#080;font-weight:bold">end</span>

Manual Allocation

Any test can be manually assigned to either of these ID types and that will take precedence:

<span class="line-numbers"><a href="#n1" name="n1">1</a></span>func <span style="color:#A60">:my_functional_test</span>, <span style="color:#606">bin</span>: <span style="color:#00D">3</span>, <span style="color:#606">softbin</span>: <span style="color:#00D">100</span>, <span style="color:#606">number</span>: <span style="color:#00D">200100</span>

The given numbers will be then be reserved and excluded from automatic assignment.

In the case where the number had already been automatically given out to a test earlier in the flow, it will be reclaimed the next time the program generator is run.

Multiple Instances of the Same Test

The assigned numbers are stored in a database by test name, such that any references to the same test name will always be assigned the same numbers.

<span class="line-numbers"><a href="#n1" name="n1">1</a></span>func <span style="color:#A60">:my_functional_test</span>
<span class="line-numbers"><a href="#n2" name="n2">2</a></span>func <span style="color:#A60">:my_functional_test</span>  <span style="color:#777"># Would be assigned the same numbers as above</span>

If a different set of numbers is desired on the 2nd occurence, then ideally this should be reflected by assigning a different test name.

If that is not possible then an index option can be used to differentiate them:

<span class="line-numbers"><a href="#n1" name="n1">1</a></span>func <span style="color:#A60">:my_functional_test</span>
<span class="line-numbers"><a href="#n2" name="n2">2</a></span>func <span style="color:#A60">:my_functional_test</span>, <span style="color:#606">index</span>: <span style="color:#00D">1</span>  <span style="color:#777"># Will be treated like a different test and assigned a different number</span>

Additionally, a :test_id option can be used to make test_ids treat differently named tests the same for the purposes of assigning numbers:

<span class="line-numbers"><a href="#n1" name="n1">1</a></span>func <span style="color:#A60">:my_func_33mhz</span>, <span style="color:#606">test_id</span>: <span style="color:#A60">:my_func</span>  <span style="color:#777"># Will all be treated like the same test by test_ids,</span>
<span class="line-numbers"><a href="#n2" name="n2">2</a></span>func <span style="color:#A60">:my_func_25mhz</span>, <span style="color:#606">test_id</span>: <span style="color:#A60">:my_func</span>  <span style="color:#777"># and will therefore all be assigned the same numbers</span>
<span class="line-numbers"><a href="#n3" name="n3">3</a></span>func <span style="color:#A60">:my_func_16mhz</span>, <span style="color:#606">test_id</span>: <span style="color:#A60">:my_func</span>

Sometimes you may even want duplicate tests to be treated the same as far as the bin number goes, but you would like them to have unique test numbers (for example).

Using specified ranges (Beta Feature)

It is assumed that in the configuration the user will configure the bins first, softbins next and numbers last

A new limited feature has been added to the plugin. It will let the user specify ranges in the flow for different TYPE (softbin/bin/number) and the plugin will figure out the next available number from that range.

Application side, the user needs to make sure that a check if in place in the interface to confirm that the softbin/bin/numbers being passed to a function or method are not nil.

Example: if number is a function of both softbin and bin, and softbin uses specific ranges, the following will be required in the configuration

Storage

The main benefit of this plugin is to get consistent number assignments across different invocations of the program generator, and for that to work it is necessary for the database to be stored somewhere. The database is a single file that is written in JSON format to make it human readable in case there is ever a need to manually modify it.

A dedicated Git repository is used to store this file, this means that it will be shared by different users of your application and they will all see a consistent view of the number allocations.

To enable Git storage create a new empty repository somewhere. This repository should be dedicated for use by this plugin and not used for anything else. It must be writable by all users of your application.

Once you have the repository, configure test_ids to use it like this:

<span class="line-numbers"><a href="#n1" name="n1">1</a></span><span style="color:#777"># This must be done before configuring</span>
<span class="line-numbers"><a href="#n2" name="n2">2</a></span><span style="color:#036;font-weight:bold">TestIds</span>.repo =  <span style="background-color:hsla(0,100%,50%,0.05)"><span style="color:#710">"</span><span style="color:#D20">ssh://git@github.com:myaccount/my_test_ids.git</span><span style="color:#710">"</span></span>
<span class="line-numbers"><a href="#n3" name="n3">3</a></span>
<span class="line-numbers"><a href="#n4" name="n4">4</a></span><span style="color:#036;font-weight:bold">TestIds</span>.configure <span style="color:#080;font-weight:bold">do</span> |config|
<span class="line-numbers"><a href="#n5" name="n5">5</a></span>  <span style="color:#777">#...</span>

The repository will then be kept up to date on every program generator invocation. A locking system will be automatically managed for you to prevent concurrent updates from multiple users.

Sometimes during development you may want to temporarily inhibit publishing to the repo, that can be achieved like this:

Notes on Duplicates

The time is recorded each time a reference is made to an ID number when generating a test program, this means that we can identify the IDs which have not been used for the longest time - for example because they were assigned to a test which has since been removed from the flow.

If the allocated range is exhausted then duplicates will start to be assigned starting from the oldest referenced ID first. This should ensure that true duplicates never happen as long as you assign an adequate range to cover the size of your test flow.

What is Not Supported

Currently if you change the configuration then all bets are off and the behavior is currently undefined. In future this plugin may support a clean up operation to assist with such changes where any IDs which are no longer valid under the new configuration get re-generated.
